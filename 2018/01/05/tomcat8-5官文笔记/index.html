一直没有通读过很多中间件的官方文档，这次恶补起来。
http://tomcat.apache.org/tomcat-8.5-doc/index.html
tomcat还在维护的有几个版本9.0,8.5,8.0,7.0
其中9.0实现了Servlet4.0规范(JavaEE8),8.5实现了Servlet3.1规范(JavaEE7),并移除BIO连接器，默认采用NIO连接器，8.0也实现了Servlet3.1规范,7.0实现了Servlet3.0规范(JavaEE 6)。
区别，可参考：https://www.zhihu.com/question/52255668
其实8.5之所以会出现，就是想在8.0与9.0之间有个稳定的新功能过度，而且8.5支持Http/2

关于怎么选择，及Servelt不同版本的区别，见:http://tomcat.apache.org/whichversion.html

不同Servlet版本的功能区别:https://en.wikipedia.org/wiki/Java_servlet#History

tomcat class loader

http://tomcat.apache.org/tomcat-8.5-doc/class-loader-howto.html
Java类加载机制采用双亲委派模型。具体不作过多介绍。在tomcat中有些不一样。

  • Bootstrap：这里的Bootstrap指得是Java类加载器模型中的Bootstrap+Ext
  • System:和java类加载器模型的APP/System加载器概念一致，不过加载路径上有些区别。java类加载器模型中的System会按照ClASSPATH环境变量的定义去加载类。而tomcat的System会忽略ClASSPATH环境变量(除非你显式定义在$CATALINA_HOME/bin/catalina.sh).此处System的加载路径为：$CATALINA_HOME/bin/bootstrap.jar，$CATALINA_HOME/bin/tomcat-juli.jar(日志相关)，$CATALINA_HOME/bin/commons-daemon.jar
  • Common:加载tomcat公共类。加载路径定义在$CATALINA_BASE/conf/catalina.properties文件中

具体每个jar包的用途解释见官文。

  • WebappX：每个tomcat实例为其中的每个web应用创建的类加载器，加载路径/WEB-INF/classes,/WEB-INF/lib.应用间相互不可见，相互独立。
    WebaapX加载器的运行规则与双亲委派机制不同，对于jre基础类，它会直接请求Bootstrap(XML解析组件是例外)，但是对于其他的，会先在本地路径进行加载，加载不到再去System->common找.
    顺序为：
  • Bootstrap classes of your JVM
  • /WEB-INF/classes of your web application
  • /WEB-INF/lib/*.jar of your web application
  • System class loader classes
  • Common class loader classes

If the web application class loader is configured with <Loader delegate="true"/>then the order becomes:

  • Bootstrap classes of your JVM
  • System class loader classes (described above)
  • Common class loader classes (described above)
  • /WEB-INF/classes of your web application
  • /WEB-INF/lib/*.jar of your web application

理解tomcat class loader

问题：为什么tomcat要自己搞一堆classcloader：
1、各个webapp中的class和lib，需要互相隔离。共享类lib可以放到CATALINE_HOME/lib下。
2、安全性问题：使用单独classloader去加载tomcat自身类库，以免被恶意破坏。
3、热部署支持

在catalinaLoader.loadClass并实例化Catalina后，为什么这里需要在其后费尽笔墨反射去setParentClassLoader呢，直接用((URLClassLoder)startupInstance).setParentClassLoader岂不是更加方便？要注意，如果这样写，这个强制转换的Catalina便会由jvm的SystemClassLoader加载进来，而startupInstance是由自己创建的URLClassLoader-catalinaLoader加载进来的，按照类的唯一性=ClassLoader+全限定类名来看并不是一个class，由此会抛一个ClassCastException。

tomcat又是如何隔离不同webapp的加载呢？
对于每个webapp应用，都会对应唯一的StandardContext，在StandContext中会引用WebappLoader，该类又会引用WebappClassLoader，WebappClassLoader就是真正加载webapp的classloader。
StandardContext隶属于Lifecycle管理，在start方法中会做一系列准备工作比如新建WebappClassLoader，另外loadOnStartup便会加载所有配置好的servlet（每个StandardWrapper负责管理一个servlet）,StandardWrapper中一个重要的方法loadServlet，InstanceManager里面的ClassLoader是StandContext的WebappLoader，于是，用该loader加载Servlet，从而控制住了Servlet中所有待加载的类。

XML Parsers and Java:
jre中的类除了JCP标准中的外，还附带了一些额外的，如XML解析API和实现(如DOM and SAX from W3C)。如果应用向使用自己的XML解析器时会造成一定的问题。如何解决呢？Java9,Java<=8的方式不一样，java9的不作介绍。java<=8支持一种叫Endorsed Standards Override Mechanism的机制来支持更新或替换非JCP标准的接口，链接里也列出了所有的可替换项。具体方式就是，将新的jar放在java.endorsed.dirs环境变量指定的目录下，多个目录用逗号分隔。默认在< java-home>/lib/endorsed下。而tomcat实现中，会在启动时自带参数-Djava.endorsed.dirs=$JAVA_ENDORSED_DIRS 当没有设定$JAVA_ENDORSED_DIRS时，会使用$CATALINA_HOME/endorsed作为默认目录(需要手动创建)，而不是< java-home>/lib/endorsed.
覆盖有风险，一定要确保100%兼容，否则会出现奇奇怪怪的错误。

SSI与CGI

服务器端内嵌（Server Side Includes，亦简称为SSI）是一种大多数仅应用于互联网上的简单解释性服务器端脚本语言。
SSI最常见的用法是将一个或多个文件的内容包含在网页服务器的页面上。例如，一张包含每日报价的页面可以通过下面一段代码将报价单包含在页面中：<!--#include virtual="../quote.txt" -->
这个包含不止局限于文件，而且还有来自程序的文本输出，或者是如当前时间的系统变量。
服务器端内嵌常用于在整个站点中包含一段公共代码，例如页眉、页脚与导航栏。使用控制代码可以有条件地将导航栏包含在内。
为了让网页服务器辨识启用了服务器端内嵌的HTML文件，从而运行这段代码。则文件名应有特殊的扩展名，默认为.shtml、 .stm和.shtm
SSI的语法相当简单：<!--#directive parameter=value parameter=value -->。这些指令都放在HTML注释区，所以若SSI功能并未启用时，除非其预览页面源代码.
在现代应用中，由于前端技术及模版引擎技术的发展，很少再用SSI了。此处不作过多介绍。详见：http://tomcat.apache.org/tomcat-8.5-doc/ssi-howto.html

CGI (Common Gateway Interface) 通用网关接口：
CGI是外部应用程序（CGI程序）与WEB服务器之间的接口标准，是在CGI程序和Web服务器之间传递信息的过程。CGI规范允许Web服务器执行外部程序，并将它们的输出发送给Web浏览器，CGI将Web的一组简单的静态超媒体文档变成一个完整的新的交互式媒体。过程如下：
⑴通过Internet把用户请求送到web服务器。
⑵web服务器接收用户请求并交给CGI程序处理。
⑶CGI程序把处理结果传送给web服务器。
⑷web服务器把结果送回到用户。
CGI可以用任何一种语言编写，只要这种语言具有标准输入、输出和环境变量。(很多都是Perl，PHP等)
详见:https://zh.wikipedia.org/wiki/%E9%80%9A%E7%94%A8%E7%BD%91%E5%85%B3%E6%8E%A5%E5%8F%A3

Connectors

http://tomcat.apache.org/tomcat-8.5-doc/connectors.html
http://tomcat.apache.org/tomcat-8.5-doc/config/http.html
分为两大类:HTTP,AJP
AJP主要用于与ApacheHttp集成做集群负载均衡的时候使用。
Tomcat8.5之后移除了对BIO连接器的支持，并默认采用NIO连接器。
支持的HTTP连接器类型:BIO(8.5版本后被移除，以前版本是默认连接器)、
NIO/NIO2、APR:http://tomcat.apache.org/tomcat-8.5-doc/config/http.html#Connector_Comparison
APR连接器：http://tomcat.apache.org/tomcat-8.5-doc/apr.html

maxThreads,maxConnections,acceptCount:
当并发请求数大于maxThreads时，server仍然会接受请求并(由Connector)创建socket连接,当达到maxConnections值(也就是所支持的最大tcp连接数,NIO模式下默认是10000.)，仍然可以继续接受socket，但是只是由操作系统放在Accept队列里，Connector并不会立即去取到Connector中,当并发请求数继续增加超过acceptCount(默认100)后，之后的请求过来时就会直接拒绝，客户端会看到”connection refused” error.

其他重要点的参数：
enableLookups ： 是否启用dns查询，默认是禁用false。如果你想使用request.getRemoteHost()可以开启，但是这对性能会有损耗，一般不建议开启。
maxPostSize：最大POST大小，默认2m，设置为-1则不限制。
port:指定tcp端口
protocol:指定使用的连接器：
org.apache.coyote.http11.Http11NioProtocol
org.apache.coyote.http11.Http11Nio2Protocol
org.apache.coyote.http11.Http11AprProtocol
如果配置值为HTTP/1.1会自动检测使用合适的连接器。

URIEncoding:如配置为UTF-8，防止GET参数含中文时乱码

nio连接器特定配置：
acceptCount默认100
compression
connectionTimeout：默认20000 ，即20s
disableUploadTimeout默认为true
executor：连接执行器(可选，亦可直接配置MaxThreads,使用executors的好处有：更详细的配置、可以配置成多个connector公用),对应Executor元素
keepAliveTimeout:与connectionTimeout时间一致，一般无需单独配置它
maxConnections：nio下默认10000
maxThreads：内部线程池大小，默认200，在指定executor后该值会被忽略。
minSpareThreads：最小的保留空闲线程数，默认10，在指定executor后该值会被忽略。
tcpNoDelay：默认为true，禁用Nagle算法，提升Http场景下的tcp传输性能。
更高级的关于tcp socket, nio,nio2,apr，ssl的配置请见文档。

conf/server.xml

1、术语

  • Server：代表整个tomcat Catalina servlet容器.默认实现为org.apache.catalina.core.StandardServer 配置：http://tomcat.apache.org/tomcat-8.5-doc/config/server.html 在server.xml中还关联了shutdown port(8005)
  • Service:中间组件，可配置多个.关联多个Connectors到一个Engine.
  • Engine：代表某个Service的请求处理管道(request processing pipeline)。一个Service可以包含多个Connectors, Engine接收并处理来自这些connectors的所有请求，并将response返回给合适的connector以便传输给客户端。
  • Host:关联network name到tomcat server, e.g. www.yourcompany.com. 一个Engine可以包含多个Host. http://tomcat.apache.org/tomcat-8.5-doc/config/host.html
  • Connector:负责处理与客户端的交互. 有两类：Http,AJP.公布Http端口(8080)或AJP端口(8009),http://tomcat.apache.org/tomcat-8.5-doc/config/http.html
  • Context:代表一个Web应用，一个Host可以包含多个Context。
